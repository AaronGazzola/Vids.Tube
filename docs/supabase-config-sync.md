# Supabase config sync (pull / diff / push)

How we keep Supabase project config under version control with the repo as the source of truth, without the dangers of `supabase config push`.

## Why this exists

Supabase has no `supabase config pull`, and `supabase config push` is **all-or-nothing**: it overwrites the *entire* remote auth config with whatever is in the local `config.toml`. Because our `config.toml` started as the stock template (`site_url = 127.0.0.1`, confirmations off, no SMTP), a blind push would wipe production settings. There is also no way to recover dashboard edits back into the file.

So instead of the native push, we use the **Management API** directly (the same API the CLI and Dashboard call) and manage an explicit, opt-in set of fields. The model mirrors Terraform:

- **`config.toml`** = desired state (committed; the source of truth).
- **`config.toml.remote`** = observed state (generated by pull; gitignored).
- The diff between them is a staging gate you review before applying.

Nothing is ever auto-written to `config.toml`, and pushes only touch fields we explicitly manage — never the rest of the remote config.

## The managed-field registry

The single source of what we manage is [`supabase/config-managed.ts`](../supabase/config-managed.ts). Each entry maps one `config.toml` value to its Management API field, in both directions, plus how to normalize it for comparison.

Currently managed (the auth slice):

| `config.toml` | Management API field (`config/auth`) | notes |
| --- | --- | --- |
| `[auth] site_url` | `site_url` | |
| `[auth] additional_redirect_urls` | `uri_allow_list` | array ↔ comma string; compared order-insensitively |
| `[auth.email] enable_confirmations` | `mailer_autoconfirm` | inverted (`enable_confirmations = !mailer_autoconfirm`) |
| `[auth.email.template.<key>] subject` | `mailer_subjects_<key>` | |
| `[auth.email.template.<key>] content_path` | `mailer_templates_<key>_content` | local file is read; remote stores inline HTML; compared by full content (shown as a sha256 + byte length) |

`<key>` ∈ `confirmation`, `recovery`, `invite`, `magic_link`, `email_change`.

To manage a new field, add one entry to the registry — every command below picks it up automatically.

## Commands

All commands need `SUPABASE_ACCESS_TOKEN` (an account-wide Management API token). It lives in Doppler (`dev_personal` config), so run everything through `doppler run --`.

```bash
# 1. Pull — read live remote config, write observed state. (read-only on remote)
doppler run -- npm run config:pull
#   writes supabase/.remote-config.json   (raw snapshot of all config endpoints)
#   writes supabase/config.toml.remote     (managed-field observed state, human-readable)

# 2. Diff — desired (config.toml) vs observed (last pull). Exits non-zero on drift.
doppler run -- npm run config:diff

# 3. Push — change remote to match config.toml. Managed fields only.
doppler run -- npm run config:push          # dry run: shows what would change
doppler run -- npm run config:push:apply    # applies, then verifies read-back
```

Email template HTML is generated separately by React Email and referenced via `content_path`:

```bash
npm run email:build   # renders emails/*.tsx -> supabase/templates/*.html
npm run email:dev     # live preview at http://localhost:3000
```

See [emails/](../emails/) and the "Email Template Development" section of `CLAUDE.md`.

## The workflow

**Change a setting (repo → remote):**
1. Edit `config.toml` (and `npm run email:build` if you changed a template).
2. `config:push` (dry) to review the diff, then `config:push:apply`.
3. `config:pull` to refresh observed state, then `config:diff` → expect `PARITY`.

**Reconcile a dashboard edit (remote → repo):**
1. `config:pull`, then `config:diff` to see what drifted.
2. Update `config.toml` to match (or decide remote is wrong and push over it).
3. `config:pull` + `config:diff` → `PARITY`.

The golden rule still applies: **one writer per setting.** A managed field is owned by `config.toml`; if someone also edits it in the Dashboard, the next `config:diff` will flag it — fix it in `config.toml`, don't keep two writers.

## What this does NOT cover

- **Secrets values** — write-only in the Management API, never readable. They stay in Doppler / env with `env()` substitution, never in `config.toml`.
- **Config `config.toml` can't express** — network restrictions, SSL enforcement, custom domains, read replicas, backups. These have their own API endpoints / CLI commands; add a dedicated managed entry or script if you want them version-controlled.
- **Schema & data** — managed by migrations (`npx supabase db push`), not this system.
- **The non-managed parts of `config.toml`** — they remain the stock template. We never run native `supabase config push`, so those defaults are inert. Don't run native `config push` — it would push those stale defaults to remote.

## Notes / limitations

- The `config.toml` parser in `config-managed.ts` is intentionally minimal (section + `key = value`); it strips `#` comments, so a managed value must not contain a literal `#`. None currently do.
- `config:diff` compares desired against the **last pull** (`.remote-config.json`), so pull before you trust a diff. `config:push` always re-fetches live remote before changing anything.
- `config.toml.remote` and `.remote-config.json` are generated artifacts and gitignored. `config.toml` is the only committed source of truth.
